# Impeccable `live` — Deep Dive vs PDLC Visual Companion

Date: 2026-04-29
Author: Claude (research subagent)
Repo at fetch time: `pbakaus/impeccable` @ `main`, Apache-2.0, 23.3k stars, default branch `main`, last push 2026-04-29
Companion-of-record: `pbakaus.impeccable.style`

---

## 1. TL;DR

- `live` is **not** a "dev preview server with hot reload." It is a full **AI design-variant generation pipeline** wired through the user's real source tree. Browser → SSE/HTTP helper server → agent long-poll → atomic multi-variant write → user accept/discard → source rewrite ("carbonization").
- Implementation footprint is large: **~330 KB across 8 scripts** plus a 46 KB reference doc. `live-browser.js` alone is **194 KB** of in-page UI (element picker, annotation canvas, variant carousel, parameter knobs, state machine, OKLCH palette, scroll-lock, Radix/Headless-UI defang).
- License is **Apache-2.0**, zero npm runtime deps (only Node built-ins). Bundles a vendored `modern-screenshot.umd.js` (29 KB) — but the bundled file's license header has been stripped, which is a license-compliance smell against the upstream MIT (`qq15725/modern-screenshot`). Anyone redistributing must restore attribution.
- Architecture is fundamentally different from PDLC's companion: PDLC serves *standalone HTML fragments authored by Muse* into an isolated frame for click-vote feedback; `live` *injects a script into the user's real running app* and rewrites the user's real source files based on AI-generated variants. They solve adjacent problems with non-overlapping mechanisms — except both run a localhost helper HTTP server.
- Top 3 capabilities PDLC's companion lacks (all in `live`): (1) **in-page element picker + annotation/strokes/comments** with screenshot upload, (2) **variant carousel with scoped CSS via `@scope`** and live parameter knobs, (3) **source-side carbonization** (atomic write of N variants into the actual source file, then rewrite on accept).
- Top 3 capabilities PDLC's companion has that `live` lacks: (1) **session-aware lifecycle bound to harness PID** (owner-PID, parent-dies-server-dies), (2) **JSONL click-event log** persisted for next-turn agent reading (`live` is purely RAM-queued via SSE/long-poll — events evaporate if no agent is polling), (3) **multi-port retry + Codex/CI platform detection** in the orchestrator shell.
- Honest verdict: **the two systems should not merge.** Different runtime model, different data shape, different audience phase (brainstorm-mockup vs construction-time visual-craft). If you want `live`'s capabilities, port it as a **separate** `scripts/live-server.cjs` (or .mjs) under a new `/pdlc craft` skill. See §6.

Confirmed by reading `live`'s code:

|---|---|---|
| **Session-aware persistence** (`.pdlc/brainstorm/<session>/` with content/, events) | ○ Partial. `live` has `.impeccable-live/annotations/session-<token>/` but only stores PNGs there — events are RAM-only and lost when the server restarts or no agent is polling. | PDLC's persistence model survives `/clear`, agent crashes, harness restarts; `live`'s does not. |

PDLC's approach is more durable. |
| **Cross-platform bash orchestrator** (`start-server.sh`) | ○ No. `live` is `.mjs`-only — runs wherever Node runs but no platform special-casing for Codex CI / Windows-Git-Bash. | PDLC's wrapper buys you platform robustness `live` doesn't have. |

| **Mermaid diagram CDN in frame** | ○ No. `live` doesn't render diagrams; it operates on the user's real running app. | PDLC's brainstorm use case is fundamentally different. |
| **Auto-exit timer** | ◐ Both have it. PDLC: 30 min idle. `live`: 8 s after no SSE clients. | `live`'s timer is much more aggressive but appropriate for its always-connected model. |

**Verdict:** PDLC's companion is genuinely better at the *brainstorm-mockup-feedback* shape: durable session state, JSONL event log, harness-PID lifecycle, platform-robust wrapper. `live` doesn't try to be that.

## 6. Recommendation: enhance vs separate

### My honest read: **Option (b) — keep both. Port `live` separately if/when needed.**

Here's why merging is wrong:

1. **Different runtime audiences.** PDLC's companion injects into a frame *PDLC controls*. `live` injects into the user's *real running app* (Vite/Next.js dev server). Merging would mean either (a) PDLC suddenly becomes opinionated about the user's framework, or (b) `live`'s capabilities get lobotomized to fit a frame-template-only world. Both are bad.

2. **Different data shapes.** PDLC events are vote-ish: `{tile, choice, timestamp}` written as JSONL. `live` events are variant-generation-ish: `{action, prompt, screenshotPath, strokes, comments, element.outerHTML, params}` flying over SSE/HTTP-POST in real time. Trying to unify those into one event bus produces an awkward superset that confuses both flows.

3. **Different lifecycle models.** PDLC's server lives ~hours, anchored to the harness PID, surviving `/clear`. `live`'s server lives ~minutes, anchored to the SSE client, dying 8 s after the browser tab closes. The lifecycle assumptions are baked into both at every layer.

4. **Different phase ownership.** PDLC's companion is a Phase-1 Inception tool (Muse during brainstorm). `live`'s capability set is a Phase-2 Construction tool — variant generation only makes sense when there's real source code to wrap and rewrite. Putting them in the same server crosses phase boundaries that PDLC otherwise enforces strictly.

### The case for option (a) — enhance in place

There is exactly one thing worth absorbing: the **screenshot capture + annotation strokes/comments** capability could enhance brainstorm. Imagine: Muse shows a mockup tile, user circles a region, types "this header feels too heavy" — that JSON-encoded annotation goes into the event log alongside the click. That's a brainstorm UX win.

But this is ~1 week of work and uses *just* the `modern-screenshot` vendor + a thin annotation overlay. You don't need `live`'s server, wrap, accept, design-parser, or carbonization. So even this absorption isn't a "merge" — it's a small targeted feature steal.

### What option (b) actually looks like

If you decide PDLC needs `live`-class capabilities, do this:

1. **New skill: `/pdlc craft`** (or extend `/build`). Lives in Phase 2.
2. **New file: `scripts/live-server.cjs`** (or `.mjs` to match impeccable). Distinct from `scripts/server.cjs`. Different lifecycle, different routes, different events.
3. **Reuse one shared piece:** the platform-detection logic in `start-server.sh`. Extract that into a shared helper so both `start-server.sh` and a future `start-live-server.sh` use the same Codex/Windows-Git-Bash branch. **This is the only refactor worth doing now**, and it costs you 1 hour.
4. **Don't bundle `live-browser.js`'s 194 KB UI.** Either link to impeccable upstream as a peer dep, or write a minimal PDLC-flavored picker (~20 KB) when you actually need one. The 194 KB is the most expensive code to maintain — it has its own MutationObserver loops, scroll-lock heuristics, Radix defang logic, OKLCH palette, state machine — and you don't need most of that for PDLC's use cases.

### The sub-option I would not pick: "extract a shared core."

Tempting because both run a localhost helper HTTP server with a port-scan and a token. But everything *above* that 100-line foundation is divergent. You'd ship a "shared core" that's mostly negative space, then maintain divergence on top. Better to just have two small servers that share nothing except a 30-line port-scan utility (which you can copy-paste; it's not worth a module boundary).

## 7. Risk + license review

### License (Apache-2.0)

- **Permits**: commercial use, modification, distribution, patent grant, private use.
- **Requires** if redistributing: include LICENSE, include NOTICE if upstream has one, state changes, preserve copyright notices. PDLC bundling any portion of `live` source must keep the Apache-2.0 header on each ported file and add a top-of-file note like "Originally from pbakaus/impeccable, modified by PDLC contributors per Apache-2.0 §4."
- **No copyleft.** You can ship modified versions under your own (compatible) license without source-release obligations.

### Vendored `modern-screenshot.umd.js` — license-compliance concern

The vendored copy at `scripts/modern-screenshot.umd.js` (29 KB) has had its license/copyright header stripped during minification. Upstream `qq15725/modern-screenshot` is **MIT** and *requires* the copyright notice be preserved in copies and substantial portions. **Impeccable upstream is currently out of compliance on its own vendored copy.** If PDLC ports this approach, restore the MIT header before shipping:

```js
/*! modern-screenshot v<version> | MIT License | https://github.com/qq15725/modern-screenshot
 * Copyright (c) qq15725 */
```

Also check upstream's NOTICE / third-party-licenses file to ensure the MIT terms are passed through. (Impeccable's repo does not appear to have a third-party-licenses doc as of fetch time.)

### Runtime dependency surface

- **Zero npm runtime deps.** All eight `.mjs` scripts use only Node built-ins (`http`, `https`, `crypto`, `fs`, `path`, `net`, `child_process`, `url`). **This is genuinely good.** Anything PDLC ports retains this property.
- **One vendored UMD blob** (modern-screenshot). MIT-permissive but see above re header.
- **One CDN dependency in production**: `live` doesn't load CDN scripts in the page. PDLC's frame-template loads Mermaid from CDN — that's a PDLC-side choice, not a `live` thing.

### Operational risks if porting

| Risk | Severity | Mitigation |
|---|---|---|
| Server runs on user's machine, listens on 127.0.0.1, accepts arbitrary file reads via `/source` | Medium — already path-traversal-protected in `live-server.mjs`. Re-audit the protection if porting. | Keep the path-prefix check; extend to a per-session allowlist if PDLC's threat model is tighter. |
| Server reads/writes user's source files (`live-wrap.mjs` / `live-accept.mjs`) | High — this *is* PDLC's Tier-2 territory (modifying source from a network-triggered event). | Wire these operations through PDLC's safety-guardrails; do not allow source rewrite without explicit user confirmation matching Tier-2 protocol. The current `live` design assumes the agent runs the wrap/accept; if you preserve that, the agent's existing tool-use confirmation flow covers it. |
| 10-MB PNG upload limit on `/annotation` | Low — DoS via repeated uploads. | Keep the 10-MB cap and add a per-session quota. |
| 8 s after-no-clients shutdown | Low — flapping if SSE disconnects briefly. | Increase to 30 s for PDLC's harness (closer to PDLC's 30-min idle but accommodating brief reconnects). |
| Token in query string — leaks via referer / proxy logs | Low (localhost only, no proxies) but worth noting. | Move to `Authorization` header on POST routes. Already low-risk because of localhost binding. |
| 194 KB `live-browser.js` is hard to audit and modify | Medium maintenance cost. | Don't port wholesale. Build PDLC's picker minimal. |
