# PDLC Deep Scan: Structural Gaps, Enhancements & Completeness Assessment

## Executive Summary

PDLC is an ambitious, well-architected Claude Code plugin that covers the full product development lifecycle with impressive depth. The vision is ~80% complete -- the core loop (init -> brainstorm -> build -> ship) is fully specified and operational, with sophisticated agent orchestration, safety guardrails, state reconciliation, and crash recovery. The gaps fall into three categories: **missing features** that the architecture implies but doesn't deliver, **structural weaknesses** in the current implementation, and **vision-level blind spots** that limit PDLC's reach.

---

## 1. Structural Gaps (Architecture promises things it doesn't deliver)

### 1.1 No `override-tier1` command exists
The guardrails hook (`hooks/pdlc-guardrails.js:33`) tells users to run `/pdlc override "<command>"` when a Tier 1 block fires, but this command doesn't exist in the router (`pdlc.md`). The double-RED confirmation protocol is fully specified in `skills/safety-guardrails/SKILL.md` (Steps 1-5), but there's no slash command or skill to actually invoke it. Users hit a dead end after the guardrail fires.

**Impact:** Tier 1 overrides are completely broken at runtime.

### 1.2 Context monitor has no way to get `used_pct`
The context monitor (`hooks/pdlc-context-monitor.js`) reads `used_pct` from a bridge file at `/tmp/pdlc-ctx-{sessionId}.json`, but nothing ever writes `used_pct` to this file. The statusline hook (`hooks/pdlc-statusline.js:148`) has a heuristic based on STATE.md mtime (age in minutes = percent), but the context monitor initializes `used_pct: 0` and never updates it from any real signal. Claude Code doesn't expose context usage to hooks.

**Impact:** The context warning system (65% WARNING, 80% CRITICAL) never fires. The progress bar in the statusline is always near 0%. The auto-checkpoint in STATE.md never triggers. This is the single biggest reliability gap -- context rot prevention was a core selling point.

### 1.3 No `bd close-all` guard implementation
CONSTITUTION.md and `safety-guardrails/SKILL.md` define a Tier 2 guard for "closing all Beads tasks at once", but `pdlc-guardrails.js` doesn't check for this pattern. The guardrails hook only inspects Bash commands, and `bd done --all` or scripted bulk closures would pass through unchecked.

### 1.4 Beads (`bd`) is an external dependency that may not exist
The entire Construction phase depends on `@beads/bd` (Beads task manager) and `dolt` (SQL database). If the npm package `@beads/bd` doesn't exist on the public registry (and I couldn't verify it does), the entire task management layer is phantom infrastructure. The `pdlc.js` installer prompts to install it, but if the package doesn't exist, init fails silently.

### 1.5 Visual companion server lacks error handling
`scripts/start-server.sh` and `scripts/server.cjs` are referenced but the server has no health check, no graceful shutdown on crash, and no fallback if the port is occupied. The brainstorm flow assumes the server starts cleanly.

### 1.6 `PDLC_PLUGIN_ROOT` vs `CLAUDE_PLUGIN_ROOT` inconsistency
The command router (`pdlc.md`) uses `${PDLC_PLUGIN_ROOT}` for path resolution, and the installer (`pdlc.js:415`) replaces this placeholder when copying commands. But `.claude/settings.json` uses `${CLAUDE_PLUGIN_ROOT}` for hooks. If Claude Code doesn't set `CLAUDE_PLUGIN_ROOT` (it's not a documented Claude Code env var), hooks fail silently.

## 2. Missing Functionality (Things the framework should have but doesn't)

### 2.1 No rollback / revert skill
The flow diagram shows a linear path from Ship -> Verify -> Reflect, but there's no defined protocol for when smoke tests fail catastrophically or a production incident occurs post-deploy. The doctor check (Check 6) can detect reverts in git history, but there's no `/pdlc rollback` command that:
- Reverts the merge commit on main
- Updates STATE.md and ROADMAP.md to reflect the rollback
- Creates an incident episode
- Triggers a decision review on what went wrong

### 2.2 No parallel feature support
PDLC is strictly single-feature. STATE.md has one "Current Feature" field, ROADMAP allows only one "In Progress" feature, and the doctor flags multiple In Progress features as CRITICAL. Real startup teams often work on 2-3 features concurrently (one in brainstorm, one in build, one in ship). The architecture would need per-feature state tracking.

### 2.3 No hotfix / interrupt protocol
There's no path for "production is on fire, drop everything and fix this bug." A hotfix flow would need to:
- Pause the current feature (save full context to STATE.md)
- Create a hotfix branch from main
- Skip brainstorm/design (emergency)
- Run a compressed build -> test -> ship cycle
- Resume the paused feature

### 2.4 No team/multi-user awareness
PDLC assumes a single Claude instance working with a single human. There's no concept of:
- Multiple engineers working different features
- Assigning tasks to specific humans
- PR review by another human (not just Claude's agents)
- Merge conflict resolution between concurrent feature branches
The doctor (Check 6) detects multi-user commits but can't act on them.

### 2.5 No metrics dashboard or trend tracking
The reflect skill captures per-episode metrics (cycle time, test pass rate, shipping streak), but there's no aggregation across episodes. No way to answer "are we getting faster?" or "which test layer fails most often?" OVERVIEW.md is append-only narrative, not structured data.

### 2.6 No dependency/package update workflow
No skill handles `npm audit fix`, dependency upgrades, or Renovate/Dependabot-style maintenance. These are common tasks that don't fit the feature lifecycle but need the same review rigor.

### 2.7 No `pdlc pause` / `pdlc resume` explicit commands
The framework has implicit resume (read STATE.md on session start), but no explicit `/pdlc pause` that cleanly saves context and no `/pdlc resume <feature>` that restores a specific paused feature. The user has to rely on auto-resume, which depends on STATE.md being accurate.

### 2.8 No feature abandonment flow
If a feature is discovered mid-build to be unviable, there's no clean way to abandon it. The deadlock protocol mentions "Abandon feature" as an option, but there's no skill that:
- Closes all Beads tasks for the feature
- Updates ROADMAP to "Dropped"
- Archives the brainstorm/design docs
- Resets STATE.md to Idle
- Records an abandonment episode

## 3. Enhancement Opportunities (Things that work but could be better)

### 3.1 Agent model assignments are aspirational
Agent files specify `model: claude-opus-4-6`, and the init skill mentions "Model override: use Haiku for Steps 1a-1e." But Claude Code slash commands and skills can't actually force model selection per-step. These annotations are documentation, not enforcement.

### 3.2 Guardrails hook only covers Bash
`pdlc-guardrails.js` fires on `PreToolUse` for `Bash` only. But destructive actions can happen through the `Edit` tool (editing CONSTITUTION.md), the `Write` tool (overwriting STATE.md), or external MCP tools. Tier 2 item 5 (modifying CONSTITUTION.md) is only caught if it's done via a bash command like `sed`, not via Claude's native Edit tool.

### 3.3 No validation of CONSTITUTION.md structure
Skills read CONSTITUTION.md for tech stack, test gates, and guardrail overrides, but there's no schema validation. If a user manually edits CONSTITUTION.md and breaks the markdown table format, the guardrails hook's `parseTier2Overrides()` silently returns empty, and test gate checks may silently pass.

### 3.4 Episode index is a flat markdown file
`docs/pdlc/memory/episodes/index.md` is appended to after each ship. For long-running projects (20+ features), searching and cross-referencing becomes unwieldy. A structured format (JSON, YAML, or at least a markdown table with consistent columns) would enable automated trend analysis.

### 3.5 MOM files accumulate without cleanup
Every party meeting, decision review, whatif analysis, and deadlock generates a MOM file in `docs/pdlc/mom/`. Over a project's lifetime, this directory becomes a dump. No archival, indexing, or pruning strategy exists.

### 3.6 The doctor could run automatically
`/pdlc doctor` is manual-only. It should optionally run as a pre-flight check at the start of every phase (lightweight mode: checks 1-4 only), surfacing state drift before it compounds.

### 3.7 No support for monorepos
PDLC assumes one project = one repo. For monorepos, there's no way to scope PDLC to a specific package or workspace. STATE.md, CONSTITUTION.md, and all memory files live at the repo root.

### 3.8 Brainstorm log and PRD are separate from design docs
The brainstorm log lives at `docs/pdlc/brainstorm/`, the PRD at `docs/pdlc/prds/`, and design docs at `docs/pdlc/design/[feature]/`. This three-way split makes it hard to find everything about a feature in one place. A `docs/pdlc/features/[feature-name]/` hierarchy would be cleaner.

### 3.9 No plugin/extension system
PDLC is monolithic. There's no way for teams to add custom skills (e.g., `/pdlc deploy-staging`), custom agents (e.g., a data engineer), or custom test layers (e.g., contract testing) without forking the repo. The "PDLC-OS Marketplace" is mentioned in the README but no extension mechanism exists.

### 3.10 The `test` skill is standalone but also embedded in build
`skills/test/SKILL.md` defines the full 7-layer test protocol, but `skills/build/steps/04-test.md` re-implements the same logic. The test skill should be the canonical source, invoked from the build step -- currently they could drift.

## 4. Vision-Level Blind Spots

### 4.1 No learning across features
PDLC captures per-feature retrospectives but doesn't learn from them. There's no mechanism to:
- Auto-suggest Constitution changes based on recurring retro items
- Detect patterns across episodes (e.g., "Phantom findings in auth modules have been accepted 3 times -- maybe add a Constitution rule")
- Feed retro improvements back into the brainstorm/build flow automatically

### 4.2 No cost/token tracking
For teams using Claude API credits, there's no visibility into how much each phase costs. Agent Teams mode spawns many subagents -- a 9-agent decision review could be expensive. No budget guardrails exist.

### 4.3 No integration with external project management
PDLC is self-contained with Beads + memory files. There's no bridge to Linear, Jira, GitHub Projects, or Notion. For teams already using these tools, PDLC creates a parallel task universe that must be manually synced.

### 4.4 No support for non-code deliverables
PDLC is code-centric. Features that are primarily configuration, infrastructure-as-code, documentation, or design-only don't fit the TDD cycle well. The TDD skill has an "infrastructure override" escape hatch, but it's a single line -- not a real alternative workflow.

### 4.5 No versioning of PDLC memory files
CONSTITUTION.md, INTENT.md, and ROADMAP.md evolve over time but have no version history beyond git. There's no diff view, no "what changed since last feature," no way to roll back a Constitution change that broke things.

## 5. Completeness Scorecard

| Dimension | Score | Notes |
|-----------|-------|-------|
| **Core lifecycle (init -> ship)** | 95% | Fully specified, deeply thought out. Minor gaps in error paths. |
| **Agent system** | 90% | Rich personas, clear responsibilities, good handoff protocol. Model override is aspirational. |
| **Safety guardrails** | 75% | Well-designed tiers but Tier 1 override is broken, Bash-only coverage is limiting, CONSTITUTION edit via Edit tool is uncaught. |
| **State management & recovery** | 85% | Excellent crash recovery with pending files and reconciliation protocol. Context monitor is non-functional (no `used_pct` source). |
| **Party mode / multi-agent** | 85% | Sophisticated orchestration with 3 modes, deadlock protocol, and durable checkpoints. Depends on Agent Teams feature availability. |
| **Developer experience** | 70% | Good install/uninstall, nice statusline, but no pause/resume commands, no rollback, no hotfix path. |
| **Extensibility** | 30% | Monolithic, no plugin system, no custom agent/skill hooks, marketplace is vaporware. |
| **Multi-user / team** | 20% | Designed for single-user single-feature. No concurrency, no task assignment, no multi-branch awareness. |
| **Observability** | 40% | Per-episode metrics exist but no aggregation, no trends, no cost tracking, no dashboard. |
| **External integrations** | 35% | GitHub only (via `gh`). No Linear, Jira, Slack, or CI/CD platform integrations. |

**Overall vision completeness: ~70%**

The core single-user, single-feature lifecycle is near-complete and impressively detailed. The gaps cluster around multi-user scenarios, extensibility, external integrations, and operational maturity (rollback, hotfix, metrics aggregation). These are the areas that separate a "personal productivity tool" from a "team development framework" -- which is what PDLC aspires to be.

## Priority Recommendations (what to plug first)

### P0 -- Broken functionality
1. **Fix Tier 1 override dead end** -- add `/pdlc override` or inline the double-RED protocol into the guardrails hook response
2. **Solve context usage tracking** -- investigate whether Claude Code exposes any context signal to hooks; if not, implement a heuristic (tool call count, token estimate from file reads) or remove the promise
3. **Verify Beads (`@beads/bd`) actually exists on npm** -- if it doesn't, the entire Construction phase is blocked

### P1 -- High-value missing features
4. **Add `/pdlc rollback`** -- revert a shipped feature with proper state cleanup
5. **Add `/pdlc hotfix`** -- compressed emergency build->ship cycle
6. **Add `/pdlc pause` and `/pdlc resume`** -- explicit state management
7. **Extend guardrails to Edit/Write tools** -- catch CONSTITUTION.md edits through any tool

### P2 -- Vision gaps
8. **Plugin/extension system** -- custom skills, agents, and test layers
9. **Cross-episode metrics aggregation** -- trend tracking in OVERVIEW.md or a dedicated metrics file
10. **Feature abandonment flow** -- clean exit from any phase
