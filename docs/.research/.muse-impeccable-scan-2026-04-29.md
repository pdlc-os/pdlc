# Muse × impeccable — Research Scan

**Date:** 2026-04-29
**Scope:** Assess whether the open-source `impeccable` repo (https://github.com/pbakaus/impeccable, Apache-2.0) can meaningfully enrich PDLC's Muse (UX Designer) agent.
**Status:** Research only. No integration work performed. No edits to `agents/muse.md` or related files.

---

## 1. Executive summary

**Verdict: GO — partial, selective, with significant adaptation work.**

`impeccable` is a substantial, well-authored, Apache-2.0 licensed Claude Code skill that gives an AI harness a directive UI/UX vocabulary. It is centered on **frontend visual craft** (typography, color, spatial design, motion, interaction patterns, responsive design, UX writing) plus a layer of **evaluation methodologies** (Nielsen heuristics scoring, cognitive-load checklist, persona-based review, AI-slop detection). It also ships a 23-command surface that mostly maps to *imperative design verbs* (`craft`, `critique`, `polish`, `clarify`, etc.).

Three key observations drive the recommendation:

1. **The reference library is the gold.** The seven topical reference files (`typography.md`, `color-and-contrast.md`, `spatial-design.md`, `motion-design.md`, `interaction-design.md`, `responsive-design.md`, `ux-writing.md`) plus `cognitive-load.md`, `heuristics-scoring.md`, `personas.md`, and `brand.md` are content-rich, opinionated, and directly applicable to Muse's review/sign-off work. PDLC currently has *zero* of this content baked into Muse — Muse's persona names principles ("low cognitive load", "accessibility", "intentional friction") but never operationalizes them with concrete thresholds, anti-patterns, or scoring rubrics.
2. **The 23 commands are mostly out-of-scope.** They presume a build-the-frontend workflow inside the skill (impeccable produces UI; PDLC's Friday produces UI). PDLC's Muse is a *reviewer and discoverer*, not a builder. Most commands (`animate`, `colorize`, `typeset`, `delight`, `overdrive`, `craft`, `live`) are skippable. A small minority (`critique`, `audit`, `clarify`, `onboard`) map to verbs Muse already does conceptually.
3. **Voice clash is real but manageable.** impeccable is *brash, declarative, "don't use ease, never #000, gradient text is banned"*. Muse is *empathetic, perceptive, composed*. The content can be carried forward, but the *delivery* must be re-voiced. Direct paste of impeccable prose into Muse's persona file would break Muse's tone.

**Recommended outcome:** create a new `agents/extensions/muse-ux-design.md` (parallel to `phantom-security-audit.md`) that absorbs roughly 60–70% of the reference-library content as Muse's domain catalog. Keep `muse.md` as the persona/voice source of truth. Optionally add a Bloom-style step in `skills/brainstorm/steps/discover/` for Muse-led design-laws audit (parallel to Phantom's `threat-model.md`). Do **not** import the 23 commands — they don't fit PDLC's lifecycle.

License (Apache-2.0) supports derivative work without attribution-required reproduction; we comply by retaining a NOTICE entry once integration happens. Research file is fine to reference impeccable directly; shipped artifacts must not.

## 2. What impeccable is

**Tagline:** "The vocabulary you didn't know you needed. 1 skill, 23 commands, and curated anti-patterns for impeccable frontend design."

**About-sidebar one-liner:** "The design language that makes your AI harness better at design."

**Thesis:** LLMs left to their own devices generate predictable, generic UIs — Inter font, purple gradients, nested cards, hero-metric templates, glassmorphism. impeccable injects an opinionated design vocabulary so AI-assisted design output stops looking AI-generated. The repo descends from Anthropic's original `frontend-design` skill and builds on it.

**Target audience:** developers using AI code assistants (Claude Code, Cursor, Gemini CLI, Codex, Pi, OpenCode, Copilot, Kiro, Trae, Rovo, Qoder — the repo ships per-harness skill bundles).

**Scope of work the skill does:** designs, redesigns, critiques, audits, polishes, animates, colorizes — i.e., it *produces and evaluates* HTML/CSS frontends. It is fundamentally a *frontend visual craft* skill, not a research/discovery/methodology skill.

**Repo shape (relevant parts):**

```
source/skills/impeccable/
├── SKILL.md                       # entry point: setup gates, design laws, command router
├── reference/                     # 35 markdown files — domain catalogs and command guides
│   ├── typography.md              # type scales, fluid type, OpenType, web-font loading
│   ├── color-and-contrast.md      # OKLCH, tinted neutrals, palette structure, dark mode
│   ├── spatial-design.md          # 4pt grid, container queries, optical adjustments
│   ├── motion-design.md           # 100/300/500 rule, easing curves, perceived perf, reduced motion
│   ├── interaction-design.md      # 8 interactive states, focus rings, modals, popover, anchor positioning
│   ├── responsive-design.md       # mobile-first, content-driven breakpoints, input-method detection
│   ├── ux-writing.md              # button labels, error templates, voice/tone, localization
│   ├── cognitive-load.md          # 8-item checklist, working-memory limits, 8 violation patterns
│   ├── heuristics-scoring.md      # Nielsen 10 scored 0–4, total 0–40 with health bands
│   ├── personas.md                # Alex/Jordan/Sam/Riley/Casey + selection framework
│   ├── brand.md                   # font selection procedure, reflex-reject list, brand bans
│   └── (24 command-specific refs: craft, critique, audit, polish, shape, clarify, ...)
├── scripts/                       # Node helper scripts (live server, screenshot, parser)
└── (per-harness mirrors at .claude/, .cursor/, .gemini/, etc.)
```

**Top-level repo:** `DESIGN.md`, `PRODUCT.md`, `AGENTS.md`, `CLAUDE.md`, `HARNESSES.md`, `NOTICE.md`, `LICENSE` (Apache-2.0), web build under `src/` and `public/`, npm package, browser extension, plus a Cloudflare-Worker-based download function.

**License:** Apache-2.0. Confirmed in repo root LICENSE and frontmatter. Permits derivative work; requires retaining copyright notice and stating significant modifications. **No copyleft. No attribution-in-output requirement.** Compatible with PDLC's borrowing model: ideas/structure carry forward, full prose paste does not, and a `NOTICE.md` line in PDLC at integration time covers the legal obligation cleanly.

**Authoring style:** brash, directive, anti-reflexive. Heavy use of imperative bans (*"Stop using HSL"*, *"Pure gray is dead"*, *"Don't use `ease`"*). Tables for thresholds and rules. Dense code blocks demonstrating the rule. A persistent "anti-AI-slop" frame: *avoid the things every LLM defaults to*. Tone is closer to a Stripe-internal design memo than a polite handbook.

### 3.1 The seven topical reference files (the core asset)

#### 3.1.1 `reference/typography.md` — HIGH FIT

**Purpose:** comprehensive typographic guidance from classical principles (vertical rhythm, modular scale, measure) through modern web type (`clamp()` fluid type, OpenType features, `text-wrap: balance/pretty`, variable fonts) to font-loading anti-shift recipes.

**Key content (verbatim-equivalent — load-bearing):**

- **Vertical rhythm:** "Your line-height should be the base unit for ALL vertical spacing." If body is `line-height: 1.5` on `16px` (= 24px), spacing values are multiples of 24px.
- **Modular scale:** 5 sizes is enough — xs/sm/base/lg/xl. Typical ratios: 1.25 (major third), 1.333 (perfect fourth), 1.5 (perfect fifth). "Pick one and commit."
- **Measure:** `max-width: 65ch`. Light-on-dark needs +0.05–0.1 line-height, +0.01–0.02em letter-spacing, optional weight-step-up.
- **Paragraph rhythm:** space between OR first-line indent. Never both.
- **Pairing rule:** "You often don't need a second font." Contrast on multiple axes when you do. Never pair two similar-but-not-identical typefaces.
- **Font loading:** `@font-face { font-display: swap }` + matching fallback metrics via `size-adjust`, `ascent-override`, `descent-override`, `line-gap-override`. Recommends Fontaine for auto-calculation. `swap` vs `optional` decision rule. Preload only the critical above-fold weight.
- **Variable fonts:** "for 3+ weights or styles" (smaller than three statics, fractional control, pairs with `font-optical-sizing: auto`).
- **Fluid type:** `clamp(min, preferred, max)`. Use for headlines on marketing/content. **Do NOT use for app UI** — "no major app design system uses fluid type in product UI." Bound: `max-size ≤ ~2.5 × min-size`.
- **OpenType feature recipes:** `font-variant-numeric: tabular-nums`, `diagonal-fractions`, `font-variant-caps: all-small-caps`, `font-variant-ligatures: none` for code. Wakamai Fondue tool reference.
- **Rendering polish:** `text-wrap: balance` for headings, `text-wrap: pretty` for prose, `font-optical-sizing: auto`, all-caps tracking 0.05–0.12em.
- **A11y:** never disable zoom; use rem/em not px; minimum 16px body; 44px+ tap targets via padding/line-height.
- **Avoid list:** more than 2–3 families, skipping fallback definitions, ignoring FOUT/FOIT, decorative fonts for body.

**Verdict:** Muse currently has none of this. Direct insert (re-voiced to Muse's tone). **Land in `agents/extensions/muse-ux-design.md` under a `### Typography` section.**

#### 3.1.2 `reference/color-and-contrast.md` — HIGH FIT

**Purpose:** color-space discipline + accessibility floor + dark-mode philosophy.

**Key content:**

- **Stop using HSL. Use OKLCH** (perceptually uniform). Reduce chroma at extreme lightness.
- **Anti-default rule:** don't reach for blue (hue 250) or warm orange (hue 60) by reflex — they're the AI defaults.
- **Tinted neutrals:** add 0.005–0.015 chroma toward the brand hue. "Pure gray is dead."
- **Palette structure table:** primary (1 color, 3–5 shades), neutral (9–11 shades), semantic (4 colors × 2–3 shades), surface (2–3 elevations).
- **Skip secondary/tertiary** unless genuinely needed.
- **60-30-10 rule** restated as visual-weight rule, not pixel-count rule.
- **WCAG table:** 4.5:1 body / 3:1 large (18px+ or 14px bold) / 3:1 UI / AAA targets.
- **Placeholder gotcha:** placeholders need 4.5:1 too — most light-gray placeholders fail.
- **Dangerous combinations:** light gray on white, gray on color, red/green, blue on red, yellow on white, thin light text on images.
- **Never pure gray or pure black:** even 0.005–0.01 chroma feels natural.
- **Dark mode is NOT inverted light mode** — different design, lighter surfaces for depth (no shadows), reduce font weight (350 not 400), desaturate accents, never pure black (12–18% lightness).
- **Token hierarchy:** primitives (`--blue-500`) + semantic (`--color-primary`). Dark mode redefines semantic only.
- **Alpha is a design smell** — heavy `rgba()`/`hsla()` = incomplete palette.

**Verdict:** Direct fit for Muse's "color contrast" review responsibility (currently a one-liner in `muse.md`). **Land in `muse-ux-design.md` under `### Color & Contrast`.**

#### 3.1.3 `reference/spatial-design.md` — HIGH FIT

**Purpose:** spacing systems, grid, hierarchy, container queries, optical adjustments, depth.

**Key content:**

- **4pt base, not 8pt** — 4/8/12/16/24/32/48/64/96. 8pt is too coarse.
- **Semantic naming:** `--space-sm` not `--spacing-8`. Use `gap`, not margins, for sibling spacing.
- **Self-adjusting grid:** `repeat(auto-fit, minmax(280px, 1fr))`.
- **Squint test** — blur the screen; can you still identify the most important element?
- **Hierarchy through multiple dimensions** table: size (3:1+ ratio), weight (bold vs regular), color (high contrast), position (top/left primary), space (surrounded by white). Best uses 2–3 dimensions at once.
- **Cards are not required** — overused. Use only when content is truly distinct, comparison-oriented, or interactively bounded. **Never nest cards inside cards.**
- **Container queries** for components — `container-type: inline-size` + `@container (min-width: 400px)`. Page = viewport queries; component = container queries.
- **Optical adjustments:** negative `margin-left: -0.05em` for left-aligned text; play icon shift right; arrows shift toward direction.
- **Touch target ≥ 44px** via padding or `::before { inset: -10px }`.
- **Semantic z-index scale:** dropdown → sticky → modal-backdrop → modal → toast → tooltip. No magic 9999.

**Verdict:** Direct fit. **Land in `muse-ux-design.md` under `### Spatial Design & Hierarchy`.** Reinforces Muse's existing "visual hierarchy" responsibility with concrete tools.

#### 3.1.4 `reference/motion-design.md` — HIGH FIT (smaller scope)

**Purpose:** duration timings, easing curves, premium motion materials, staggering, reduced motion, perceived performance.

**Key content:**

- **100/300/500 rule** table — 100–150ms (instant feedback), 200–300ms (state changes), 300–500ms (layout changes), 500–800ms (entrances). Exit ≈ 75% of enter.
- **Don't use `ease`.** Specific cubic-bezier curves named:
  - ease-out (entering): `cubic-bezier(0.16, 1, 0.3, 1)`
  - ease-in (leaving): `cubic-bezier(0.7, 0, 0.84, 0)`
  - ease-in-out (toggles): `cubic-bezier(0.65, 0, 0.35, 1)`
  - quart out (default): `cubic-bezier(0.25, 1, 0.5, 1)`
  - quint, expo variants.
- **Avoid bounce/elastic** — "trendy in 2015 but now feel tacky."
- **Premium motion materials beyond transform/opacity:** blur, backdrop-filter, saturation/brightness, shadow bloom, SVG filters, masks, clip paths, gradient-position. Hard rule: "avoid animating layout-driving properties casually (`width`, `height`, `top`, `left`, margins)."
- **Staggered animations:** `animation-delay: calc(var(--i, 0) * 50ms)` with `style="--i: 0"`. Cap total stagger time.
- **Reduced motion:** `@media (prefers-reduced-motion: reduce)` — vestibular disorders affect ~35% of adults over 40. Provide alternatives, not just disable.
- **Perceived performance:** 80ms threshold (anything under feels instant). Active vs passive time. Optimistic UI for low-stakes only. "Too-fast responses can decrease perceived value."

**Verdict:** Strong fit. **Land in `muse-ux-design.md` under `### Motion Design`.** Muse currently doesn't speak to motion at all — gap closed.

#### 3.1.5 `reference/interaction-design.md` — HIGHEST FIT

**Purpose:** interaction-state coverage, focus rings, forms, modals, popovers, dropdown positioning, destructive actions, keyboard navigation, gesture discoverability.

**Why it matters most for Muse:** this is the file most aligned with Muse's *current* responsibility list — async states, error messaging, focus indicators, touch targets are all things `muse.md` already names but doesn't operationalize.

**Key content:**

- **8 interactive states table:** default / hover / focus / active / disabled / loading / error / success — every interactive element needs all of them designed. The common miss: hover without focus, or focus without hover. **Keyboard users never see hover.**
- **Focus rings:** never `outline: none` without replacement — accessibility violation. Use `:focus-visible` for keyboard-only. 3:1 contrast min, 2–3px thick, offset from element.
- **Forms:**
  - Placeholders aren't labels.
  - Validate on blur, not keystroke (exception: password strength).
  - Errors below field, `aria-describedby`-connected.
- **Loading states:** optimistic updates for low-stakes only. **Skeleton screens > spinners.**
- **Modals:** `inert` attribute on background, native `<dialog>` with `showModal()`, automatic focus trap.
- **Popover API:** `popovertarget` / `popover` for tooltips/dropdowns/non-modal overlays. Light-dismiss free, top-layer stacking, no z-index wars.
- **Dropdown positioning:** the `position: absolute` inside `overflow: hidden` clipping bug = "the single most common dropdown bug in generated code." Solution: CSS Anchor Positioning API (`anchor-name`, `position-anchor`, `position-area`, `@position-try`) — Chrome 125+. Fallback: Popover API + portals + `getBoundingClientRect()` + `position: fixed`.
- **Destructive actions: undo > confirm.** "Users click through confirmations mindlessly." Confirmation only for truly irreversible / high-cost / batch operations.
- **Roving tabindex** for tab/menu/radio groups.
- **Skip links** (`<a href="#main-content">`).
- **Gesture discoverability:** swipe-to-delete is invisible — partially reveal, coach marks first use, always provide a visible fallback.

**Verdict:** Highest fit. The 8-state table alone would dramatically sharpen Muse's current decision-checklist item #3 ("loading, error, empty, success") by extending it to 8 named states. **Land in `muse-ux-design.md` under `### Interaction Design`.**

#### 3.1.6 `reference/responsive-design.md` — MEDIUM FIT

**Purpose:** mobile-first methodology, breakpoints, input-method detection, safe-area insets, responsive images, layout adaptation, testing.

**Key content:**

- Mobile-first with `min-width` queries. Content-driven breakpoints, not device-targeted (typical: 640/768/1024).
- Detect *capabilities* not *device*: `pointer: fine` vs `pointer: coarse`, `hover: hover` vs `hover: none`. "Don't rely on hover for functionality."
- Safe-area insets: `env(safe-area-inset-top)` etc., requires `viewport-fit=cover` meta.
- Responsive images: `srcset` width descriptors + `sizes` declaration. `<picture>` for art direction.
- Layout adaptation patterns (nav: hamburger → compact → full; tables → cards via `data-label`; `<details>/<summary>` for mobile collapsibles).
- Testing: emulators don't capture real touch, CPU, network, font rendering, browser chrome. Minimum: one real iPhone, one real Android, tablet if applicable.
- Avoidances: desktop-first, device detection over feature detection, separate codebases.

**Verdict:** Useful but partially overlaps with what Friday/Bolt would handle on the implementation side. **Land in `muse-ux-design.md` under `### Responsive Design` as a shorter section** — focus on the input-method-detection rule and the mobile-first review check, leave implementation specifics.

#### 3.1.7 `reference/ux-writing.md` — HIGH FIT

**Purpose:** copy quality across button labels, error messages, empty states, voice/tone, accessibility writing, localization, terminology, confirmations, loading, forms.

**Key content:**

- **Button labels:** verb + object. Generic ban table (OK → Save changes / Submit → Create account / Yes → Delete message / Cancel → Keep editing / Click here → Download PDF).
- **Destructive labels:** "Delete" not "Remove" (suggests reversibility). Include affected count: "Delete 5 items".
- **Error message formula:** What happened → why → how to fix.
  - Bad: "Invalid input"
  - Good: "Email address needs an @ symbol. Example: user@example.com"
- **Error templates by scenario** (format, missing, access, connectivity, server failure).
- **Tone rule:** system limitation, not user failure. "Please enter a date in MM/DD/YYYY format" not "You entered an invalid date."
- **Empty states = onboarding moments:** acknowledgment + value prop + action pathway.
- **Voice vs tone:** voice = consistent brand, tone = situational. Table by context (success / error / async / destructive confirm).
- **Never use humor for errors.**
- **A11y in writing:** link text functional alone ("View pricing plans" not "Click here"). Alt text describes information not mechanics. Icon buttons require `aria-label`. Decorative `alt=""`.
- **Localization expansion ratios:** German +30%, French +20%, Finnish +30–40%, Chinese −30%. Separate numeric values from text. Spell out terms. Sentence-level atomic strings.
- **Terminology table:** Delete (not Remove/Trash), Settings (not Preferences/Options), Sign in (not Log in/Enter), Create (not Add/New). Maintain a glossary.
- **Copy redundancy:** "Say it once, say it well."
- **Loading messaging:** specific ("Saving your draft...") + duration expectation for long waits.
- **Confirmation dialog rule:** most are design gaps; prioritize undo. When needed: name the action, name the consequence, action-specific buttons not yes/no.
- **Form instructions:** placeholder demonstrates format; explain *why* for non-obvious fields.

**Verdict:** Direct fit. Muse's persona names "humane defaults" and "intentional friction" but doesn't operationalize copy quality at all. **Land in `muse-ux-design.md` under `### UX Writing`.** This may be the single most leverage-positive import — it's the lowest-cost-to-adopt and shows up in every shipped feature.

### 3.2 The methodology files

#### 3.2.1 `reference/cognitive-load.md` — HIGH FIT

**Purpose:** assessment framework based on three load categories (intrinsic / extraneous / germane), 8-item checklist, working-memory limits, 8 violation patterns.

**Key content:**

- **Three loads:** intrinsic (task, only structure), extraneous (design failures, eliminate), germane (learning, support).
- **8-item checklist** — users should be able to: complete tasks without distraction; process info in ≤4-item groups; identify visually grouped elements; recognize screen priority; focus on one decision at a time; choose from ≤4 visible options per decision; not need to reference previous screens; access complexity progressively.
- **Scoring:** 0–1 failures = acceptable; 2–3 = moderate; 4+ = urgent intervention.
- **Working memory: ≤4 items** (revised Miller). Practical limits: 5 nav, 4 form fields per section, 1 primary + 2 secondary buttons, 4 dashboard metrics, 3 pricing tiers.
- **8 violation patterns:** Wall of Options, Memory Bridge, Hidden Navigation, Jargon Barrier, Visual Noise, Inconsistent Patterns, Multi-Task Demands, Context Switching — each with the corrective.

**Verdict:** This is gold for Muse's review output. Currently `muse.md` mentions "low cognitive load" as a taste-profile item without ever defining it. **Land in `muse-ux-design.md` under `### Cognitive Load Assessment`.** Add the 8-item checklist as a Muse decision-checklist extension.

#### 3.2.2 `reference/heuristics-scoring.md` — HIGH FIT

**Purpose:** Nielsen's 10 usability heuristics with a 0–4 scoring rubric, total 0–40, mapped to health bands.

**Key content:**

- 10 heuristics (visibility of system status / match real world / user control / consistency / error prevention / recognition over recall / flexibility & efficiency / aesthetic minimalism / error recovery / help & documentation), each scored 0–4 with criteria.
- **Health bands:** 36–40 excellent / 28–35 good / 20–27 acceptable / 12–19 poor / 0–11 critical.

**Verdict:** This is the single most structurally valuable import. Also good fit for `skills/review/` UX-review portion.

### 3.4 SKILL.md (the entry-point file)

**Purpose:** entry point that loads context (PRODUCT.md, DESIGN.md), enforces setup gates, declares design laws, routes by command.

**Key content:**

- 6 setup gates (context / product / command / craft / image / mutation).
- Design laws (color, theme, typography, layout, motion).
- **Absolute bans (match-and-refuse):** side-stripe borders, gradient text, glassmorphism as default, hero-metric template, identical card grids, modal as first thought.
- **Copy principles:** "Every word earns its place. No restated headings, no intros that repeat the title." "No em dashes."
- **AI Slop Test:** designs must not be identifiable as AI-generated.

**Verdict:** Setup gates and routing are skill-orchestration plumbing, not relevant. **The "Absolute bans" and "AI Slop Test" are highly relevant** — they encode anti-pattern detection that Muse should run during review. **Land the bans as a checklist inside `muse-ux-design.md` under `### Anti-Patterns to Refuse` (rendered as Muse's voice — empathetic but firm).**

## 4. Thematic findings

Cross-cutting patterns observed across the impeccable corpus:

### 4.1 Anti-reflexive philosophy

The guiding meta-principle is "the AI defaults are wrong because they're defaults." Hue 250 (blue) and hue 60 (warm orange) are banned not because they're bad colors but because they're the *reflex*. Same with Inter, Plus Jakarta Sans, Cormorant, glassmorphism, gradient text. The skill's job is to *prevent the LLM's natural drift toward the median*. This is a philosophy PDLC's Muse should absorb — Muse's persona already names "convention is a gift to the user," but doesn't name "AI-default convention is the enemy of distinctive product."

### 4.2 Directive voice with concrete thresholds

Every recommendation comes with a number, a name, or a code snippet:
- `≥1.25 ratio` between scale steps
- `65–75ch` line measure
- `100/300/500ms` durations
- `4pt` spacing base
- `≤4` items in working memory
- `44×44px` touch target
- `oklch(12-18%)` dark mode background

This is the opposite of Muse's current persona, which talks in feelings ("intuitive flows", "low cognitive load") without thresholds. Importing impeccable's *thresholds* would make Muse's review *measurable* and *citable*. Phantom already has this discipline (CVSS scores, EOL dates) — Muse currently doesn't.

### 4.3 Eight-state thinking

Where Muse currently asks for 4 states (loading/error/empty/success), impeccable asks for 8 (default/hover/focus/active/disabled/loading/error/success). The expansion catches the keyboard-only-user blind spot (focus state) and the disabled-affordance blind spot — both are real review gaps.

### 4.4 Scoring rubrics

Two scoring frameworks recur: Nielsen 10 (0–4 each, 0–40 total) and the audit 5-dimension scorecard (0–4 each, 0–20 total). Both convert subjective UX review into a *trendable number*. This pairs naturally with PDLC's METRICS.md — UX score per feature → trend over releases → tech-debt visibility on UX dimensions specifically.

### 4.5 Tools-as-vocabulary

References to specific tools (Wakamai Fondue, Fontaine, WebAIM Contrast Checker, Polypane, MDN dialog/popover/inert) embed the *correct toolchain* into the recommendations. Muse currently names no tools. Carrying the tool references forward gives the team concrete next steps.

### 4.6 What's missing from impeccable

A few things Muse needs that impeccable does NOT have:

- **Journey mapping / Jobs-to-be-Done** — impeccable is screen-by-screen, not journey-level. Muse already does multi-screen flow review during UX Discovery; impeccable wouldn't add to that.
- **PRD/BDD scenario completeness review** — Muse's existing decision-checklist item #7 (Given/When/Then scenario coverage) has no impeccable analog. Impeccable doesn't talk about specifications, only artifacts.
- **Mental-model vocabulary** — Muse's central frame ("user's mental model" / "what does the user believe is about to happen") is not in impeccable. Impeccable is more about visual craft than cognitive psychology.
- **Existing-product-pattern grounding** — PDLC's UX Discovery grounds in `[ui-inventory]` (shipped components/decisions). Impeccable is greenfield-oriented; it doesn't reason about consistency with previously-shipped features in the same product.

These four are Muse's existing strengths and impeccable doesn't replace them.

## 8. Risks and open questions

### 8.1 License — green-flagged, with one obligation

Apache-2.0 permits derivative work, modification, sublicensing, and distribution. The legal obligation is: retain the copyright notice + state significant modifications. We satisfy this by adding a single line in `NOTICE.md` (or creating one if it doesn't exist) along the lines of *"Portions of `agents/extensions/muse-ux-design.md` are derived from `pbakaus/impeccable` (Apache-2.0). Significant modifications: re-voiced for Muse's persona, re-scoped to PDLC's lifecycle context, structural reorganization, addition of PDLC-specific personas."* The user's "no impeccable references in shipped artifacts" rule applies to the *content* of agent files — the NOTICE file is a legal-compliance file, not a shipped agent artifact, and is the right place for the attribution. **Open question for user: do you want the NOTICE.md line, or do you want me to claim derivative-by-idea (no verbatim) and skip attribution? The first is unambiguously safe; the second is defensible if no verbatim text crosses over.**

### 8.2 Voice clash

impeccable is brash, declarative, Tyler-Durden-of-design. Muse is empathetic, perceptive, composed. The content can be ported but the *delivery* must change. Two specific things to watch:

- **Imperative bans** ("Stop using HSL", "Pure gray is dead", "Don't use `ease`") need to become *Muse-voiced soft directives* ("Use OKLCH wherever the team can — perceptual uniformity matters when you're trying to communicate hierarchy through lightness alone").
- **Anti-AI-slop framing** is a great philosophical frame but it's snarky. Muse can carry the *substance* — refuse the AI-default patterns — without the *snark* — describing them as "patterns I'll flag because they don't help your user understand what's happening" rather than "AI slop."

Failure mode if voice isn't adapted: Muse's persona becomes incoherent — empathetic in one section, brash in another. Voice consistency check is part of the integration QA.

### 8.3 Scope creep

The seven reference files are long. Pasting them all *as-is* would balloon `muse-ux-design.md` to 3–4× the size of `phantom-security-audit.md`, which is already long. Mitigation: the extension is meant to be Muse's *catalog* — Phantom's extension is also large and that's accepted. But the trim work (especially in `responsive-design.md` where Friday's content overlaps) is real and should be done deliberately.

### 8.4 Coverage in PDLC's lifecycle context

impeccable is a *visual craft* skill. PDLC has lifecycle phases (init/brainstorm/build/ship). impeccable's content has to be tagged to *when* in the lifecycle each piece applies:

- Typography / color / spatial / motion / interaction / responsive / UX writing → review during Construction (Muse's review of Friday's PR) and Operation (verify during ship).
- Cognitive load / Nielsen heuristics / persona scan → review during Construction; can also run during Inception's UX Discovery (if a flow is already mocked).
- Anti-patterns to refuse → review during Construction (sign-off gate).

The integration plan needs to wire each new section to the lifecycle moment(s) where it applies. Phantom's extension does this implicitly via "review-time" framing; Muse's extension should do the same explicitly.

### 8.5 Things I couldn't determine without asking

- **Whether the user wants impeccable's tool references** (Wakamai Fondue, Fontaine, WebAIM Contrast Checker, Polypane, etc.) to live in PDLC. They're useful but they're also an opinionated tool list. Phantom's extension has a "Tooling reference" section — we could mirror that pattern.
- **Whether `agents/muse.md` should be updated to point to the extension**, or whether the extension is silently auto-loaded when Muse is invoked (the same way `phantom-security-audit.md` extends `phantom.md`). Looking at `phantom-security-audit.md`, the pattern is: extension file says "Extends: `agents/phantom.md`" and the precedence rule. `phantom.md` itself doesn't appear to point back to the extension (and PDLC's loading logic auto-applies it). We'd follow the same one-way-link pattern.
- **Whether to expose any of this as a skill** (e.g., `skills/muse-ux-review/SKILL.md`) for invocation outside the construction-review path. Probably not worth it for v1.
- **Whether to allow Muse to reference the impeccable repo in its review output** as a "see also" link for the user. The user's "no impeccable references in shipped artifacts" rule says no — but that might cost the user the ability to deep-link readers to canonical sources. Worth confirming the rule applies even to external "further reading" links.

### 9.1 The reflex-reject font list (verbatim)

Fraunces · Newsreader · Lora · Crimson · Playfair Display · Cormorant · Syne · IBM Plex · Space Mono · Inter · DM Sans · Outfit · Plus Jakarta Sans · Instrument Sans

### 9.2 The 8 interactive states (table)

| State | When | Visual Treatment |
|-------|------|------------------|
| Default | At rest | Base styling |
| Hover | Pointer over (not touch) | Subtle lift, color shift |
| Focus | Keyboard/programmatic focus | Visible ring (3:1 contrast min) |
| Active | Being pressed | Pressed in, darker |
| Disabled | Not interactive | Reduced opacity, no pointer |
| Loading | Processing | Spinner, skeleton |
| Error | Invalid state | Red border, icon, message |
| Success | Completed | Green check, confirmation |

### 9.3 Easing curves (verbatim CSS values)

```css
--ease-out-quart: cubic-bezier(0.25, 1, 0.5, 1);   /* default */
--ease-out-quint: cubic-bezier(0.22, 1, 0.36, 1);
--ease-out-expo:  cubic-bezier(0.16, 1, 0.3, 1);
/* Entering */                cubic-bezier(0.16, 1, 0.3, 1);
/* Leaving */                 cubic-bezier(0.7, 0, 0.84, 0);
/* Toggles */                 cubic-bezier(0.65, 0, 0.35, 1);
```

### 9.4 Cognitive-load 8-item checklist

Users should be able to:
1. Complete primary tasks without distraction
2. Process information in ≤4-item groups
3. Identify visually grouped related elements
4. Immediately recognize screen priority
5. Focus on one decision sequentially
6. Choose from ≤4 visible options per decision
7. Avoid referencing previous screens
8. Access complexity progressively

Scoring: 0–1 failures acceptable / 2–3 moderate / 4+ urgent.

### 9.5 Nielsen scoring health bands

- 36–40: Excellent (minor polish)
- 28–35: Good (address weak areas)
- 20–27: Acceptable (significant improvements needed)
- 12–19: Poor (major overhaul required)
- 0–11: Critical (redesign needed)

### 9.6 Severity tags

P0 blocks task completion / P1 significant difficulty / P2 annoyance with workarounds / P3 polish.

### 9.7 Absolute bans (impeccable's match-and-refuse list)

1. Side-stripe borders (colored `border-left`/`border-right` >1px)
2. Gradient text (`background-clip: text`)
3. Glassmorphism as default
4. Hero-metric template (big number + label + stats + gradient)
5. Identical card grids (repeated icon + heading + text)
6. Modal as first thought (exhaust inline alternatives)

### 9.8 The 100/300/500 motion rule

| Duration | Use Case |
|---|---|
| 100–150ms | Instant feedback (button press, toggle) |
| 200–300ms | State changes (menu open, tooltip) |
| 300–500ms | Layout changes (accordion, modal, drawer) |
| 500–800ms | Entrance animations (page load, hero reveals) |

Exit ≈ 75% of enter.

### 9.9 Color: never pure

- Pure gray (`oklch(50% 0 0)`) and pure black (`#000`) don't exist in nature
- Even chroma 0.005–0.01 toward brand hue is enough
- Tinted neutrals create subconscious cohesion with brand color

### 9.10 The 5 personas (impeccable's, for reference — to be replaced with PDLC-flavored versions)

- **Alex** — impatient power user
- **Jordan** — confused first-timer
- **Sam** — accessibility-dependent (screen reader, keyboard-only)
- **Riley** — deliberate stress tester
- **Casey** — distracted mobile user

Field structure: Profile / Behaviors / Test Questions / Red Flags.

### 9.11 The audit 5-dimension scorecard

- A11y (0–4) — WCAG compliance
- Performance (0–4) — layout thrashing, animation eff., assets, render
- Theming (0–4) — token usage, dark mode, color consistency
- Responsive (0–4) — fixed widths, touch targets, breakpoints, overflow
- Anti-Patterns (0–4) — AI-slop tells
- Total 0–20 with health bands (18–20 excellent / 14–17 good / 10–13 acceptable / 6–9 poor / 0–5 critical).

### 9.12 The 8 cognitive-load violation patterns

1. **Wall of Options** → Group and prioritize
2. **Memory Bridge** → Maintain visible context
3. **Hidden Navigation** → Show location via breadcrumbs/active states
4. **Jargon Barrier** → Use plain language with inline definitions
5. **Visual Noise** → Establish hierarchy: primary/secondary/muted
6. **Inconsistent Patterns** → Standardize similar actions
7. **Multi-Task Demands** → Sequence steps sequentially
8. **Context Switching** → Co-locate decision-relevant information
