## 1. Guiding principles

Three constraints govern the hybrid integration. They take precedence over any specific borrowing decision.

**Principle 1 — Audience flip.** impeccable is authored for a *human operator*: the user types `/critique` or `/audit` at the harness CLI and the skill responds. Muse is an *agent inside PDLC*, summoned by other agents or by the orchestration layer, running methodology *internally* during ideation and review. Every piece of borrowed content must be re-framed so the *agent* is the operator, not the user. The 23 impeccable commands are the most affected: they do not become PDLC commands. The underlying checklists become Muse's internal procedures invoked at lifecycle moments, not user-typed verbs.

- Impeccable framing: "Run `/critique` to get a Nielsen scorecard."
- PDLC framing: "During Construction Review, Muse scores the heuristics and reports the total against the health bands."

**Principle 2 — PDLC's concept is non-negotiable.** The 4-phase lifecycle, the 9-agent team, the file-based memory model, the Constitution-overrides-defaults rule, the tier-1/2/3 safety model, the interaction-mode discipline (Sketch vs Socratic), the Beads task model, the Party Review and Decision Review patterns, the Pitch Round + Vote — none of these change to accommodate borrowed content. impeccable's content conforms to PDLC's shape, never the reverse. Anything that clashes with PDLC's structure gets either dropped or rewritten until it fits.

**Principle 3 — Re-voice, don't paste.** impeccable's voice is brash, declarative, anti-AI-slop ("Stop using HSL," "Pure gray is dead," "Don't use `ease`," "Modal as first thought"). Muse's voice is empathetic, perceptive, composed, insightful. Direct paste breaks Muse's persona. The rule:

- **Retain verbatim:** thresholds, code snippets, table values, easing-curve values, OKLCH values, ms durations, percentage thresholds, named heuristic terms (e.g., "Nielsen 10"). These are load-bearing specificity — the reason for borrowing in the first place.
- **Re-voice surrounding prose:** ban-style imperatives → Muse-voiced soft directives; user-facing addressing → agent-internal narration; PDLC terms over impeccable terms (e.g., "the team" not "you"; "during Construction Review" not "when you run /critique").

§6 has a re-voicing playbook with concrete before/after pairs.

**Goal these principles serve:** PDLC gets a meaningfully richer UX *ideation and build* experience — Muse becomes a UX agent with operationalized methodology, not just a persona that names principles — *without* losing the structural and tonal coherence that makes PDLC PDLC.

## 2. The two-layer model

| Layer | Where it lives | When it loads | What it does |
|---|---|---|---|
| **A — Catalog** | `agents/extensions/muse-ux-design.md` | Auto-loaded whenever Muse is summoned, regardless of phase | Muse's reference vocabulary: thresholds, scorecards, checklists, anti-patterns, persona red-flag scan. Read-only knowledge. Same shape as Phantom's `phantom-security-audit.md`. |
| **B — Workflow gates** | `skills/{phase}/steps/{step}.md` + `templates/{artifact}.md` | Triggered at specific lifecycle moments based on triage | Procedures that USE Layer A. Each gate has triage tiers, produces an artifact, and feeds findings into Decision Review and METRICS. |

**Why two layers:** Layer A is *what Muse knows*; Layer B is *what Muse does at each lifecycle moment*. Mixing them produces an unreadable extension file. Phantom's precedent — catalog in the extension file, gate as a separate skill step (`skills/brainstorm/steps/threat-model.md`) — is exactly the model Muse follows. Muse benefits more than Phantom from the separation because Muse has more lifecycle moments where her work fires.

## 4. Layer B — the workflow gates

Five gates total. Each specifies *trigger, triage, procedure, artifact, downstream wiring*.

### 4.1 Brainstorm Step 7 (UX Discovery) — extension, not new step

**File touched:** `skills/brainstorm/steps/discover/07-ux-discovery.md` (existing, Muse-led).

**What changes:** add two prompts to the existing UX Discovery flow that Muse runs as part of ideation:

- *Cognitive-load 8-item self-assessment* — Muse rates each item against the proposed flows; ≥2 failures get logged as discovery findings.
- *Persona red-flag pre-scan* — Muse names 2–3 PDLC personas relevant to the feature's user base (chosen from §3 catalog item 11) and walks their red-flag list against the proposed flows.

**Triage:** features without UI surface skip both prompts (`[ui-inventory]` value of "no UI" short-circuits the step extension).

**Artifact:** findings appended to the existing UX Discovery summary. No separate file.

**Downstream wiring:** if cognitive-load failures ≥4 *or* any persona red-flag is "blocking," Muse raises a Decision Review trigger before progressing to Define.

**Why an extension, not a new step:** UX Discovery already exists and is Muse-led. The new prompts are deepenings of an existing flow, not a new gate. A new step here would inflate the brainstorm sequence unnecessarily.

### 4.2 Brainstorm Step 10.6 — Design Laws Audit (NEW)

**File created:** `skills/brainstorm/steps/discover/10.6-design-laws-audit.md` (Muse-led, parallel to Phantom's `skills/brainstorm/steps/threat-model.md`).

**Trigger:** runs after Step 10.5 threat-modeling, before Step 11 plan. Conditional on triage outcome.

**Triage gate (3 yes/no questions, mirroring threat-modeling triage):**

| Question | If yes |
|---|---|
| Does this feature add or modify any user-facing UI surface? | required for Lite or Full |
| Does this feature introduce a new flow, page, or significant interaction pattern? | escalates to Full |
| Does this feature touch first-experience pathways (onboarding, first-time empty state, signup, install)? | escalates to Full |

| All three "no" | One "yes" (Q1 only) | Two or three "yes" |
|---|---|---|
| **Skip** — pure backend or no-UI feature; record skip in Define notes and proceed | **Lite** — anti-pattern scan + UX writing pass + 8-state coverage check | **Full** — Nielsen 0–40 + Audit 5-dim 0–20 + cognitive-load + persona red-flag + anti-patterns + UX writing + 8-state |

**Procedure (Full mode — Lite is a strict subset):** four phases mirroring Phantom's threat-modeling.

- **Phase A — Setup.** Muse loads the proposed flows from the design package and the catalog. Consults `[ui-inventory]` to ground in shipped patterns.
- **Phase B — Heuristic scoring.** Muse scores Nielsen 10 (0–4 each), sums to 0–40, places against health bands. Each ≤2 score becomes a finding with severity tag (P0/P1/P2/P3).
- **Phase C — Operational checks.** 8-state coverage on every interactive element; cognitive-load 8-item checklist; persona red-flag scan against PDLC personas.
- **Phase D — Anti-pattern scan.** Muse runs the catalog's "Anti-Patterns to Refuse" against the proposed design. Hits become P0 or P1 findings.
- **Phase E — Cross-talk and human approval.** Findings presented in a Design-Laws Roundtable (Party Review pattern). Pitch Round + Vote applies if there is severity disagreement. P0 findings block progress; P1–P3 can be accepted-as-tradeoff with an ADR.

**Artifact:** `templates/ux-review.md` (NEW — see §5). Lives at `docs/pdlc/features/<feature>/ux-review.md` per feature.

**Downstream wiring:**
- P0 findings block brainstorm → Plan; resolved via redesign or `/pdlc override`.
- P1–P3 findings get logged as ADRs in `DECISIONS.md` (mitigate later / accept) under the same model Phantom uses for accepted-risk threats.
- The scorecard total + finding counts become the *initial* values carried into Construction's UX Review and tracked in METRICS.md.

**Where Muse and Phantom interact:** Steps 10.5 (Phantom-led) and 10.6 (Muse-led) run sequentially, not in parallel. Phantom's mitigations may force Muse's design hand (e.g., a required auth confirmation modal Muse would otherwise call AI-slop). Phantom's threat-model artifact references UX implications; Muse reads them as input to her audit. *Edge case:* if the proposed design is fundamentally broken, Muse should bounce work back to Define before Phantom invests effort — see §8 open question 3.

**Interaction-mode wiring:** in Sketch mode, Muse runs the audit and reports findings concretely. In Socratic mode, Muse conducts Phase E by surfacing findings as questions ("Do you think a keyboard-only user has a working path through this flow?"). Catalog content does not change between modes; only the *conduct* of Phase E does.

### 4.3 Construction Review — Muse cites the scorecard

**File touched:** `skills/build/party/spawn-and-mom.md` (existing Party Review skill).

**What changes:** when Muse contributes to Party Review on a feature that ran Step 10.6, Muse:

- Compares the *design-time* scorecard (from Step 10.6) against an *as-built* scorecard she runs on the implementation.
- Flags deltas (e.g., implementation is missing 3 of 8 interactive states the design committed to).
- Findings flow into the existing Pitch Round + Vote and Decision Review escalation.

**Triage:** Lite/Full at Step 10.6 → as-built scorecard required at Construction Review. Skip at Step 10.6 → no scorecard required (Muse still does her current Party Review contribution per `agents/muse.md`).

**Artifact:** the as-built scorecard appends a new *"As-Built Audit"* section to the same `ux-review.md` from Step 10.6. One file per feature, tracking design-time → as-built deltas.

**Downstream wiring:** delta drives a Reflect-phase metric — does the as-built consistently lose points against the design? That's a process signal worth surfacing in METRICS.md.

### 4.4 Ship Verify — UX verification step (NEW)

**File created:** `skills/ship/steps/ux-verify.md` (Muse-led, parallel to `skills/ship/steps/fix-lint.md`).

**Trigger:** runs inside `/pdlc ship` Verify, after smoke tests and security verify, before Reflect.

**Triage (lighter than 10.6 — this is a final pre-deploy check, not full ideation):**
- **Skip** — feature had no UX scorecard (Step 10.6 was skipped).
- **Lite** — feature had a UX scorecard. Muse runs a final UX writing pass + anti-pattern sweep + 8-state spot-check against the as-deployed artifact. ~5-minute pass.

**Procedure (Lite):** Muse loads UX Writing + Anti-Patterns to Refuse + 8-state coverage from the catalog. Compares against as-deployed UI. Flags drift.

**Artifact:** appends a *"Ship Verify"* section to the same `ux-review.md`. P0 deltas (anti-pattern that snuck in, copy that says "An error occurred") block the ship; P1+ get an ADR.

**Downstream wiring:** ship-time findings update METRICS.md and trigger a Reflect-phase callout — what slipped through design-time *and* as-built reviews?

### 4.5 Reflect — METRICS.md integration

**Files touched:** `templates/METRICS.md` (existing) + `skills/ship/steps/reflect.md` (existing).

**What changes:** add a *UX scorecard trend* metric category. For every feature that ran Step 10.6, METRICS.md tracks:
- Nielsen total at design-time / as-built / ship-verify
- Audit scorecard total at the same three points
- Cognitive-load failure count at the same three points
- Number of P0/P1/P2/P3 findings carried as ADRs in `DECISIONS.md`

**Trend signals to surface:**
- Are scorecards trending up or down across features? (Capability building vs eroding.)
- Is the design-time → as-built delta consistently negative? (Design over-promising or build under-delivering.)
- Are P1+ findings accumulating in `DECISIONS.md` without ever closing? (UX tech debt building up.)

**Downstream wiring:** existing Reflect retrospective adds a UX-scorecard agenda item. Muse contributes the trend reading.

## 6. Re-voicing playbook

The single most important non-mechanical task in the integration. Specificity is what made impeccable a candidate for borrowing — voice changes, numbers don't.

| Source (impeccable) | PDLC re-voiced (Muse's voice, PDLC terms) |
|---|---|
| "Stop using HSL. Pure gray is dead." | "OKLCH is preferred for product UI — its perceptual uniformity makes lightness-driven hierarchy more reliable. Tinted neutrals (chroma 0.005–0.015 toward the brand hue) feel more cohesive than pure greys." |
| "Don't use `ease`. Use these specific curves." | "Default `ease` rarely fits product motion — it overshoots on entrances. Muse recommends explicit easing tokens: `ease-out-quart` for entrances, `ease-in-quart` for exits, `ease-in-out-quint` for toggles." |
| "Modal as first thought is wrong. Exhaust inline." | "Muse refuses to sign off on a modal as the *first* response to a UX problem. Modals trap attention; inline alternatives (tooltips, popovers, expandable sections, dedicated routes) usually preserve user agency. Muse's review will ask: what makes this modal-shaped, not inline-shaped?" |
| "Run `/critique` and you'll get a Nielsen scorecard." | "During Construction Review, Muse scores the heuristics and reports the Nielsen total against the health bands. Findings with score ≤2 carry a severity tag (P0/P1/P2/P3) and feed into Party Review's findings list." |
| "Pure black (`#000`) is dead." | "Pure black is rarely the right choice for product UI. Even `oklch(12-18%)` lightness with a small chroma value reads as black to users while preserving the depth and warmth of a real palette." |
| "AI Slop Test: designs must not be identifiable as AI-generated." | "Muse runs an *AI-default detection* pass — flagging the patterns LLMs reflexively reach for (purple-blue gradient hero, glassmorphism cards, hero-metric template, identical-card grids, gradient text on `background-clip: text`). The team can keep any of these patterns deliberately, but Muse will not let them through *by reflex*." |
| "Side-stripe borders banned." | "Side-stripe borders (a colored `border-left`/`border-right` >1px on a card) are an AI-default pattern. Muse flags them on review. The team can override with rationale; the default disposition is decline." |
| "Use this checklist to audit cognitive load." | "Muse's cognitive-load assessment runs the 8-item checklist below; failures count toward the cognitive-load score recorded in the UX review artifact." |

**Retention rule (verbatim):** thresholds, code snippets, table values, easing-curve cubic-bezier values, OKLCH values, ms durations, percentage thresholds, the Nielsen 10 heuristic *names* — all stay verbatim. The *rationale prose around them* gets re-voiced.

**Lossy-paraphrase risk:** if re-voicing strips specificity (e.g., "use `cubic-bezier(0.16, 1, 0.3, 1)` for entrances" → "use a soft easing curve"), the import is no longer load-bearing. Specificity is the entire reason for borrowing. *Voice changes; numbers don't.*

**Term substitutions (PDLC dialect):**

| impeccable term | PDLC term |
|---|---|
| "you" / "the user runs" / "the operator" | "Muse" / "the team" / "during {phase} {step}" |
| "the skill" | "the catalog" / "Muse's review" |
| "/{command}" (e.g., `/critique`) | "{phase} {gate}" (e.g., "Construction Review") |
| "design output" / "the design" | "the proposed flows" / "the as-built artifact" / "the shipped feature" (per lifecycle moment) |
| "AI slop" | "AI-default patterns" |
| "the absolute bans" | "the anti-patterns Muse refuses" |

## 8. Open questions specific to the hybrid plan

1. **Triage criteria precision.** The 3 yes/no questions in §4.2 are first-draft. Real-world calibration (e.g., does a backend API change with a tiny admin-tool tweak qualify as "UI surface"?) needs ground-truth from running it on a few features. Suggest: implement Wave 2 with the questions as specified, refine after the first 3 features run through it.

2. **METRICS schema vs template version.** `templates/METRICS.md` has a versioned schema (`pdlc-template-version`). Adding the UX scorecard trend section is a versioned migration — bump the template version and use the existing migration path in `pdlc upgrade`.

3. **Cross-talk between Phantom and Muse.** When threat-modeling and design-laws audit both fire on the same feature (the common case for any new UI), order matters. Default: Phantom 10.5 → Muse 10.6. Edge case: when the proposed design is so off-spec that threat-modeling would waste effort, Muse should bounce work back to Define before Phantom invests. Suggest: §4.2 spec includes a Muse "is this design coherent enough to threat-model?" pre-check inline at the end of Step 10.5.

4. **Sketch / Socratic mode wiring.** PDLC's interaction-mode discipline affects how Step 10.6 is conducted. In Sketch, Muse reports findings concretely. In Socratic, Phase E surfaces findings as questions. The catalog content does not change; the *conduct* of the gate does. The §4.2 spec includes this — needs validation against the live interaction-mode skill once integration starts.

5. **Framework citations in the catalog.** Phantom defers to OWASP/CWE/etc. in `phantom-security-audit.md` as canonical authorities. Muse's catalog references real tools but does not cite *frameworks* the same way (no "WCAG 2.2," "ARIA Authoring Practices," "Material 3 motion guidelines"). Question for the user: should the catalog explicitly cite WCAG/ARIA/etc. as Muse's canonical accessibility authority? This makes Muse's authority more legible to the team.

1. **Dogfood on PDLC itself.** Run the full integration on a real PDLC feature before declaring v1 stable. Retroactively running Step 10.6 on the Atlas rename or a similar shipped feature gives us ground truth on triage criteria and scorecard fit. Also a useful artifact for the wiki — a worked example.
2. **`/pdlc doctor` extension.** Add 4 UX-aware checks to the existing 8:
   - `agents/extensions/muse-ux-design.md` exists and is non-empty
   - `templates/ux-review.md` template version matches plugin version
   - `templates/METRICS.md` has the "UX scorecard trend" section (template version current)

5. **Pre-merge smoke test per wave:**

### 9.4 Failure-mode-and-recovery

If a wave breaks something post-release (regression discovered after the fact):

- **Hotfix path:** standard `/pdlc hotfix` on PDLC itself — emergency compressed build-ship for the regression.
- **Rollback path:** `pdlc upgrade --to v2.X.Y` to the prior version. User data preserved.
- **Constitution opt-out:** if the regression is "this gate is too aggressive," users add a Constitution override locally. Captures the failure without forcing a rollback for everyone.
- **Doctor surfaces drift:** if the regression silently broke template alignment, `/pdlc doctor` will flag it next run. We rely on this as the long-tail safety net.

## 10. Visual companion vs `live`-class server — architecture and unified URL

This section captures the decision from the deep-dive at `docs/.research/.impeccable-live-deep-dive-2026-04-29.md` and addresses the seam between the two servers' URLs.

### 10.1 Decision: keep both servers separate

PDLC's existing visual companion (`scripts/server.cjs` + `start-server.sh` + `frame-template.html`) and any future `live`-class server stay as two independent backends, not merged into one. Four reasons (full reasoning in deep-dive §6):

1. **Different runtime audiences.** The visual companion serves *standalone HTML fragments authored by Muse* into a frame PDLC controls. A `live`-class server injects a script into the user's *real running app* (Vite/Next.js/etc.). Merging means either (a) PDLC becomes opinionated about the user's framework, or (b) `live`'s capabilities get lobotomized to fit a frame-only world. Both bad.
2. **Different data shapes.** Visual companion events are vote-ish: `{tile, choice, timestamp}` written as JSONL. `live`-class events are variant-generation-ish: `{action, prompt, screenshot, strokes, comments, element.outerHTML, params}` over SSE/HTTP-POST in real time. Unifying them produces an awkward superset.
3. **Different lifecycle models.** Companion: harness-PID-bound, hours, survives `/clear`. `live`-class: SSE-client-bound, minutes, dies 8s after browser tab closes. Lifecycle assumptions are baked in at every layer.
4. **Different phase ownership.** Companion is Phase-1 Inception (brainstorm-time mockup voting). `live`-class is Phase-2 Construction (variant generation against real source). Putting them in one server crosses phase boundaries PDLC otherwise enforces.

### 10.2 Today's narrow exception — targeted feature borrowing

One capability worth absorbing into the existing visual companion *today*: **screenshot capture + annotation strokes/comments**. Use case: Muse shows a mockup tile during brainstorm; user circles a region and types "this header feels too heavy"; that JSON-encoded annotation goes into the JSONL event log alongside the click vote. Brainstorm UX win.

- **Effort:** ~1 week.
- **What's needed:** vendor `modern-screenshot` (MIT — restore the license header per the deep-dive's compliance note), add a thin annotation overlay (~30 KB SVG-stroke + comment-pin UI) to `frame-template.html`, add a `POST /annotation` endpoint to `server.cjs` with disk storage under `$STATE_DIR/annotations/`, extend the JSONL event payload schema.
- **What's NOT needed:** `live`'s server, source-wrap machinery, accept/carbonization, design parser, full 194 KB UI. This is a targeted feature import, not a port.

This work belongs in a later wave (post Wave 4) — call it Wave 7 — once the core integration is stable. It is independently shippable.

### 10.3 Future `/pdlc craft` skill — when and what

If/when PDLC moves into Phase-2 visual iteration (variant generation from real source code), this is when a `live`-class server makes sense. Triggers for considering it:

- Users repeatedly ask for "show me three variants of this component"
- Construction Review starts wanting variant comparisons that go beyond what Muse can render in a fragment
- The team is doing enough greenfield UI work that source-rewriting from browser feedback would meaningfully accelerate them

If those conditions hit, the implementation:

1. **New skill: `/pdlc craft`.** Lives in Phase 2 (Construction). Muse-Friday joint lead — Friday writes source, Muse reviews variants.
2. **New file: `scripts/live-server.cjs`** (or `.mjs` to match impeccable). Distinct from `scripts/server.cjs`. Different lifecycle, different routes, different events.
3. **Reuse the platform-detection helper** (see §10.5 for the refactor). The only piece worth sharing between the two servers.
4. **Don't bundle the 194 KB `live-browser.js`.** Either link to impeccable upstream as a peer dep, or write a minimal PDLC-flavored picker (~20 KB) when actually needed. The 194 KB has its own MutationObserver loops, scroll-lock heuristics, Radix defang logic, OKLCH palette, full state machine — most of which PDLC doesn't need at v1.
5. **License obligations:** Apache-2.0 attribution comments per ported file; restore MIT header on the vendored `modern-screenshot.umd.js` (impeccable upstream is currently out of compliance on this).

This is a future effort, **not** part of the Muse integration plan. Listed here so that when it does happen, the constraints from this decision are captured.

### 10.4 The unified URL — PDLC visual portal at `http://localhost:7352/`

The architecture decision (two servers) creates a seam: brainstorm gives the user one URL, `/pdlc craft` would give a different URL. Across a feature's lifecycle the user could be asked to copy-paste two URLs into their browser at different phases. That breaks PDLC's "one consistent integrated flow" promise.

**Chosen solution: a stable PDLC visual portal at `http://localhost:7352/`.**

Port `7352` = **"PDLC" on a numeric phone keypad** (P=7, D=3, L=5, C=2). Memorable, distinct from common dev-server ranges, and a small piece of brand identity baked into the dev experience. Users who type the URL from memory get it right on the first try.

The portal is a thin proxy server that:

- Is bookmarkable. User opens it once at install time and never copies a URL again.
- Reads an "active visual session" manifest from disk (`<project>/.pdlc/active-visual.json`).
- Proxies *all* HTTP / WebSocket / SSE traffic to the currently active backend (visual companion OR `live`-class server, whichever is up).
- When the active backend changes (brainstorm ends, craft starts), the proxy follows. The user's tab stays open. URL never changes. Content updates when the new backend renders.
- When no backend is active, shows a friendly idle page ("No active PDLC visual session. Start one via `/pdlc brainstorm` or `/pdlc craft`.").
- If multiple backends are up simultaneously (both brainstorm and craft active for different features), shows a small chooser; default = most-recently-active.

**Pros:**
- Genuinely seamless UX. One URL, forever, across phases.
- Both backends remain unchanged; architecture decision intact.
- Bookmarkable; surfaces in browser autocomplete.
- Memorable port via keypad mnemonic — easier to recall than any random high port.

**Cons:**
- One more long-lived process to manage (~150-line Node script).
- WebSocket / SSE proxying has subtleties (connection upgrade handling, keepalive, backpressure). Node's built-in `http.createServer` + `net` socket forwarding handles it; or use `http-proxy` if we accept the dependency.
- Manifest synchronization: both backends must write to / clear the manifest reliably on start/stop/crash. Edge case: harness reaps a backend without graceful shutdown. Mitigation: the portal stats the manifest on each request and falls back to the idle page if the referenced backend fails a health check.

**Effort:** ~3–4 days. ~150-line proxy script + manifest write/clear in `start-server.sh` (and future `start-live-server.sh`) + an idle fallback page.

#### Complementary access patterns (alongside the bookmarkable URL)

These layer on top of the portal to serve different user habits — they don't replace it.

- **Status-bar surfacing.** PDLC's status bar shows `http://localhost:7352/` whenever any visual session is active. Click-to-copy. For users who want to glance-confirm. ~30 lines.
- **`pdlc open` CLI.** New command that launches the portal URL in the user's default browser. For keyboard-driven users who don't bookmark, and for "I lost my tab, where was that URL?" moments. ~20 lines.

Together: bookmark for browser-tab users, status bar for glance-and-confirm, `pdlc open` for keyboard users. Full coverage at low marginal cost on top of the portal itself.

#### Backend lifecycle and coexistence

The portal does not require killing one backend to start another. Both can run simultaneously; the portal multiplexes which one the user currently sees. PDLC does not need to issue an explicit shutdown call to the visual companion when the craft server starts.

**Default flow — brainstorm → craft transition:**

1. **State 1 — Brainstorm only.** Visual companion on port 52341 (random high port). Manifest at `<project>/.pdlc/active-visual.json` = `{active: "brainstorm", port: 52341, secondary: []}`. Portal proxies `7352 → 52341`.
2. **State 2 — User runs `/pdlc craft`.** Craft server starts on port 8400 (new process — visual companion stays alive). Manifest updates: `{active: "craft", port: 8400, secondary: [{kind: "brainstorm", port: 52341}]}`. Portal switches active proxy target to 8400. Pushes SSE event to all connected clients (see next subsection). User's tab reloads, lands on craft content at the same URL `http://localhost:7352/`.
3. **State 3 — Brainstorm idles out.** The visual companion's existing 30-min idle auto-exit handles cleanup. We tighten this on phase transitions: when the brainstorm phase completes, the visual companion enters **grace mode** with a 10-min idle timer. After grace expires — or when the harness dies — the visual companion exits cleanly and the manifest's `secondary` entry is removed.

**Coexistence guarantees:**

- Multiple backends can be alive at once. The portal does not enforce uniqueness.
- Backends do not need to communicate with each other. They only write/clear their own manifest entry.
- If a backend dies ungracefully (crash, SIGKILL), its manifest entry can become stale. Mitigation: the portal stats the manifest on each request and verifies the referenced backend responds to a health check; on failure, it falls back to the next-most-recent active session or the idle page.
- When the harness dies (Claude Code session ends), all backends die via owner-PID binding. The manifest is cleared by the last-process-out, or rebuilt fresh on next session.

**Phase transition signal:** PDLC's existing phase-transition hooks fire when brainstorm completes. The new lifecycle adds one small action — signal the visual companion to drop its idle timer to grace mode (10 min). Implementation: a `POST /lifecycle/grace` endpoint on the visual companion, called by the brainstorm-phase-complete hook. Equivalent endpoint on the future `live`-class server.

#### Browser auto-reload via injected SSE script

The portal must tell the browser when to reload — otherwise the user's existing tab stays loaded from the prior backend even after the manifest swaps.

**Mechanism:** the portal injects a tiny script into every HTML response it serves on the way out (~20 lines). Pattern is similar to impeccable's `live-inject.mjs` script-tag injection, but minimal — only auto-reload, no UI.

```html
<script>
  // Injected by PDLC visual portal — auto-reloads on backend swap.
  (function () {
    const es = new EventSource('/portal/events');
    es.onmessage = function (e) {
      const msg = JSON.parse(e.data);
      if (msg.type === 'backend-changed') location.reload();
    };
  })();
</script>
```

**Portal-side responsibilities:**

- Watch the manifest with `fs.watch` on `<project>/.pdlc/active-visual.json`.
- On manifest change (or backend health check failure), broadcast `{type: "backend-changed", to: "<new-active-kind>"}` to all connected `/portal/events` SSE clients.
- Inject the script before `</body>` in HTML responses only (`Content-Type: text/html`). Non-HTML (JSON, JS, images, screenshots) passes through untouched.
- Maintain the connected-clients list in memory (small — typically 1–2 connections).
- Reserve the `/portal/*` URL namespace; never proxy these paths to backends.

**Edge cases:**

- **Strict CSP on the proxied backend.** If the new backend's HTML carries a CSP that blocks inline scripts, the portal's injected script won't execute. Fallback: serve the script from `/portal/auto-reload.js` and inject `<script src="/portal/auto-reload.js">` — the portal's own origin (`localhost:7352`) is allowed by most CSPs that match `script-src 'self'` for content served by the portal itself. If the backend's CSP forbids it entirely, document the limitation and fall back to a manual-refresh banner.
- **WebSocket / SSE traffic from the underlying backend.** The portal's `/portal/events` SSE is a separate channel from any SSE/WS the backend runs. They coexist on the same proxy via different paths. The portal must reserve `/portal/*` and pass everything else through.
- **Reload during user input.** If the user is mid-interaction (annotating a mockup, typing in a form) when the manifest changes, an immediate `location.reload()` loses the input. Mitigation: the injected script shows a small banner ("Switching to craft view in 5s — [Cancel] [Reload now]") and only auto-reloads after the timer elapses. Small UI complexity, much better data preservation. Suggest implementing the banner from day one.

#### Look-back and multi-feature handling

Three escape hatches for when the default "active = most-recently-active" doesn't match what the user wants:

1. **`pdlc open --brainstorm` / `pdlc open --craft`** — CLI explicitly switches the portal's active proxy to the named backend. Triggers the same SSE-driven reload as a phase transition. Most-recently-used flag updates so future implicit access stays on the chosen backend until the next override.
2. **Portal chooser overlay** — when multiple backends are listed in the manifest's `secondary` array, the portal injects a small chooser into the wrapper page (corner pill: "Active: craft · Also: brainstorm [switch]"). One click swaps the active backend and triggers a reload. Hidden when only one backend is active. ~40 lines of HTML/JS injected alongside the auto-reload script.
3. **Direct URL** — the underlying backend's URL (`http://localhost:52341/`) still works for the lifetime of the server. Power users can bookmark or visit it directly. The portal does not intercept or hide direct backend access.

**Multi-feature scenario.** If the user has two features in flight (feature A in brainstorm, feature B in craft), both backends run with separate session directories under `.pdlc/brainstorm/<A>/` and (future) `.pdlc/craft/<B>/`. The portal's manifest tracks both; the chooser overlay surfaces both with feature labels: "Active: craft (feature-B) · Also: brainstorm (feature-A)". Default is most-recently-touched; user can pin via `pdlc open --feature <name>` or the chooser. This works because the manifest is per-project (not per-feature) — multiple features within one project share the same portal but distinct active sessions in `secondary`. PDLC's existing "one feature in active development at a time" convention keeps this from getting unwieldy, but the data model supports the edge case.

#### When this ships

The portal does not depend on the Muse integration — it's a visual-companion infrastructure improvement that benefits brainstorm even before any `/pdlc craft` exists. It can ship in parallel with Wave 2, or earlier. Suggest making it Wave 0 of the broader visual-craft initiative so the seamless URL is in place before the second backend ever exists.

### 10.5 Implementation sketch (the refactor that's worth doing now)

One small refactor is worth doing during Wave 1 of the Muse integration:

**Extract `start-server.sh`'s platform-detection logic** into `scripts/lib/platform-detect.sh`. Both the existing `start-server.sh` and any future `start-live-server.sh` source it. Codex CI / Windows-Git-Bash branching lives in one place.

### 10.6 Open questions specific to the URL UX

1. **Default port — confirmed: 7352** ("PDLC" on a numeric phone keypad: P=7, D=3, L=5, C=2). Avoids conflicts with the existing companion's random-port range (52000+) and impeccable's `live` default (8400). Expose `PDLC_PORTAL_PORT` env var for users who already have something on 7352. Document the choice and mnemonic in CLAUDE.md and the wiki so it's discoverable.
2. **Multi-project handling.** If the user has two PDLC projects open in two terminals, both try to spawn a portal on 7352 → conflict. Two reasonable resolutions: (a) **one portal per machine** — the second project detects the existing portal and registers its manifest with it; the portal becomes a project-multiplexer with a project chooser when more than one is active; (b) **per-project portals** — first project gets 7352, subsequent projects fall back to 7353, 7354, etc. Option (a) preserves the bookmark contract for the user's most common case (one project at a time) and is the recommended default. Document fallback behavior either way.
3. **Persistence of the bookmark across `pdlc upgrade`.** The portal URL must be stable across PDLC version upgrades. The contract: **the PDLC portal URL is `http://localhost:7352/` and will not change without a major version bump.** Document this in CLAUDE.md and the wiki so users have an explicit guarantee for their bookmarks.
4. **Authentication.** The portal proxies to backends that may have token-based auth (impeccable's `live` does). The portal needs to either (a) inject the token transparently after reading the manifest, or (b) be auth-passthrough and the user's bookmark is just `http://localhost:7352/` with the token attached on each redirect. Probably (a) — keeps the user's bookmark clean.
5. **Whether the existing visual companion's URL stays stable as a "direct" URL.** Some users may want the unmodified backend URL (e.g., for debugging). Suggest: portal does both — direct URL still works (`http://localhost:52341/`), but the portal URL is the canonical one we surface in status bar / CLI / docs.
6. **Grace-mode timer length.** Suggested 10 min after brainstorm-phase-complete before the visual companion auto-exits. Should this be configurable via Constitution? Pro-configurability: users with slow workflows want longer; users with tight machines want shorter. Pro-fixed: simpler model, fewer knobs to tune. Suggest: ship at fixed 10 min, add `PDLC_VISUAL_GRACE_MINUTES` env var only if real users complain.
7. **Reload-during-input behavior — banner mandatory or opt-in?** The injected auto-reload script can either reload immediately on manifest change (loses any in-flight user input) or show a 5-second cancel-able banner. Suggest banner from day one — input loss is a worse failure mode than a small UI element. The banner adds ~30 lines to the injected script. Confirm before implementing.
